How EdgeEval turns a quantized model
into a deployment decision

A computer-vision model evaluation pipeline purpose-built for edge deployment: compare a baseline FP32 model against a quantized INT8, ONNX, or RKNN export, and get a per-class regression report instead of a single misleading accuracy number.

01

Upload baseline & candidate

Provide your original FP32 model (.pt) and the quantized or optimized export you want to validate (.onnx, INT8), plus a YOLO-format validation set (images/ + labels/).

02

Inference on identical data

Both models run against the exact same images, so any accuracy or latency difference is attributable to the optimization itself — not dataset variance.

03

IoU matching & per-class mAP

Predicted boxes are matched to ground truth at an IoU of 0.5. Precision, recall, and AP are computed per class, not just as one aggregate score.

04

Latency benchmark

Warm-up runs are discarded, then both models are timed on identical input on the same hardware — isolating the real speed/accuracy tradeoff of quantization.

05

Threshold-based flags

Any per-class AP drop beyond a configurable tolerance is flagged WARNING or CRITICAL — a deterministic, CI-style gate instead of a subjective read of a chart.

06

Deployment recommendation

A rule-based verdict — safe to deploy, or not — backed by the full structured report so you can see exactly which class regressed and by how much.

Why per-class evaluation matters for edge AI quantization

Quantizing a YOLO or other CV detection model to INT8 for RKNN, ONNX Runtime, or another edge runtime almost always trades some accuracy for speed — but that tradeoff is rarely uniform across classes. A model can lose 15–20 points of AP on one class (say, person) while barely moving on others, and the aggregate mAP can shift by only a percentage point or two, hiding a regression that would fail badly in production on exactly the class your deployment depends on.

  • Aggregate mAP@0.5 alone cannot tell you which class regressed.
  • Precision and recall are tracked separately — a security camera use case cares about recall (missed detections), a quality-control line cares about precision (false alarms).
  • Latency and accuracy are reported together, because the real question is never "is it faster," it's "is it still accurate enough at this speed."
  • Thresholds are configurable and rule-based, so the pass/fail gate behaves like a CI test — deterministic, repeatable, and explainable.
